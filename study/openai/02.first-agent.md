## LLM은 생각보다 훨씬 단순한 존재다

지금까지 ChatGPT나 Claude 같은 모델을 사용하면서 은연중에 가지고 있었던 착각이 하나 있었다.

나는 LLM이 무언가를 "할 수 있다"고 생각했다.

예를 들어 날씨를 알려주거나, 계산을 하거나, 파일을 분석하거나, 이메일을 보내는 모습을 보면서 모델이 실제로 그런 작업을 수행하는 줄 알았다.

하지만 이번 강의를 들으면서 깨달은 것은, LLM은 생각보다 훨씬 제한적인 존재라는 것이다.

LLM은 기본적으로 텍스트를 생성하는 모델이다.

조금 충격적이지만,

LLM은 결국

> 지금까지의 대화를 보고 다음에 나올 가능성이 가장 높은 텍스트를 예측하는 시스템

이다.

물론 이 과정이 엄청나게 복잡하고 강력하기 때문에 마치 사고하는 것처럼 보이지만, 근본적으로는 텍스트 예측이라는 사실은 변하지 않는다.

그래서 LLM은 스스로 코드를 실행할 수 없다.

인터넷에 접속할 수도 없고,

데이터베이스를 조회할 수도 없고,

파일을 열 수도 없고,

사용자를 대신해서 행동할 수도 없다.

그저 텍스트를 생성할 뿐이다.

---

## 그렇다면 AI Agent는 무엇인가?

강의를 듣기 전에는 AI Agent가 더 똑똑한 GPT라고 생각했다.

하지만 지금은 Agent를 전혀 다른 관점으로 보게 되었다.

Agent는 새로운 모델이 아니다.

Agent는 구조(Architecture)다.

LLM 위에 여러 가지 시스템을 추가해서 현실 세계와 상호작용할 수 있도록 만든 구조가 Agent이다.

즉,

```text
LLM
+
Tools
+
Runtime
+
Memory
```

가 합쳐졌을 때 비로소 Agent가 된다.

여기서 핵심은 LLM 자체는 변하지 않는다는 것이다.

LLM은 여전히 텍스트를 생성할 뿐이다.

달라지는 것은 그 주변에 있는 시스템이다.

---

## Function Calling을 배우며 바뀐 관점

이번 강의에서 가장 중요한 개념은 Function Calling이었다.

처음 이름만 들었을 때는

"아, 모델이 함수를 호출하는 기능이구나."

정도로 생각했다.

하지만 실제로는 훨씬 더 흥미로운 개념이었다.

Function Calling은 함수를 실행하는 기능이 아니다.

Function Calling은

> 모델이 어떤 함수를 실행해야 하는지 결정하는 기능

이다.

이 차이가 매우 중요하다.

예를 들어 사용자가

```text
베를린 날씨 알려줘
```

라고 말했을 때,

나는 예전에는 모델이 내부적으로 날씨 함수를 실행한다고 생각했다.

하지만 실제로는 그렇지 않다.

모델은 단지 이렇게 말하는 것이다.

```json
{
  "name": "get_weather",
  "arguments": {
    "city": "Berlin"
  }
}
```

즉,

> "개발자님, 제가 보기에 get_weather 함수를 city='Berlin'으로 실행하면 될 것 같습니다."

라고 제안하는 것이다.

실제로 함수를 실행하는 것은 Python이다.

이 부분을 이해하고 나니 Agent의 구조가 훨씬 명확하게 보이기 시작했다.

---

## LLM의 역할은 Planner다

이번 강의를 들으면서 가장 마음에 들었던 표현은

> LLM은 Planner이고 Python은 Executor이다.

라는 관점이다.

예전에는 LLM이 모든 것을 하는 것처럼 느껴졌다.

하지만 실제 구조를 들여다보면 역할이 명확하게 나뉘어 있다.

### LLM의 역할

* 사용자의 의도를 이해한다.
* 어떤 도구가 필요한지 판단한다.
* 어떤 인자를 넣어야 하는지 결정한다.
* 결과를 보고 최종 답변을 작성한다.

### Python Runtime의 역할

* 함수를 실행한다.
* API를 호출한다.
* 데이터를 가져온다.
* 파일을 읽는다.
* 결과를 반환한다.

즉 LLM은 관리자이고,

Python은 실제 작업자에 가깝다.

이 관점으로 보니 Agent Framework들이 왜 존재하는지도 이해되기 시작했다.

---

## Memory에 대한 오해

이번 강의에서 또 하나 흥미로웠던 것은 Memory에 대한 부분이다.

평소에는 ChatGPT가 무언가를 기억한다고 생각했다.

하지만 실제로는 전혀 다르다.

Agent는 기억하지 않는다.

우리가 기억한다고 느끼는 이유는 매 요청마다 이전 대화를 다시 전달하기 때문이다.

즉,

```python
messages = [...]
```

가 사실상 Agent의 기억이다.

사용자의 질문,

모델의 답변,

Tool 실행 결과,

이 모든 것이 messages 안에 저장된다.

그리고 다음 요청이 올 때마다 이 전체 내용을 다시 모델에게 전달한다.

그래서 Memory의 본질은

"저장"이라기보다

"재생(Replay)"

에 가깝다.

이 부분을 이해하고 나니 LangGraph가 왜 Memory를 별도의 개념으로 다루는지도 조금씩 이해되기 시작했다.

## Tool Result가 중요한 이유

처음에는 Tool을 실행하면 끝이라고 생각했다.

하지만 실제 Agent Loop를 구현해보니 Tool Result가 훨씬 중요했다.

모델은 Tool 실행 결과를 자동으로 알지 못한다.

예를 들어

```text
User:
베를린 날씨 알려줘
```

↓

```text
LLM:
get_weather를 호출해주세요
```

↓

```text
Python:
Berlin is sunny
```

이렇게 끝나면 모델은 아직 날씨 결과를 모른다.

그래서 Tool Result를 messages에 추가해야 한다.

```python
{
    "role": "tool",
    "content": "Berlin is sunny"
}
```

그 순간부터 모델은

> "아, 내가 요청했던 함수의 결과가 Berlin is sunny였구나."

를 이해할 수 있게 된다.

그리고 나서야

```text
현재 베를린은 맑은 날씨입니다.
```

라는 최종 답변을 만들 수 있다.

이 구조를 이해하면서 Agent가 단순한 챗봇이 아니라는 것을 처음으로 체감했다.

## 지금까지 만든 것은 무엇인가?

Weather Bot을 만들 때는 단순한 예제라고 생각했다.

하지만 지금 다시 보니 그것은 사실 가장 작은 형태의 Agent였다.

사용자가 질문한다.

↓

모델이 어떤 Tool을 사용할지 결정한다.

↓

Python이 Tool을 실행한다.

↓

결과를 모델에게 돌려준다.

↓

모델이 최종 답변을 생성한다.

이것이 바로 Agent Loop이다.

그리고 앞으로 배우게 될 LangGraph, OpenAI Agents SDK, MCP 같은 것들은 결국 이 루프를 더 편리하고 강력하게 만들기 위한 도구들이다.

---

## 이번 강의를 통해 얻은 가장 중요한 통찰

Function Calling은 함수를 호출하는 기술이 아니다.

Function Calling은 LLM이 현실 세계와 상호작용할 수 있도록 만드는 첫 번째 단계이다.

LLM은 여전히 텍스트 생성기일 뿐이다.

하지만 Tool과 Runtime이 추가되는 순간,

LLM은 단순한 ChatBot이 아니라 행동(Action)을 계획하는 시스템이 된다.

그리고 Agent란 결국,

```text
Reasoning
↓
Action
↓
Observation
↓
Reasoning
```

이라는 루프를 반복하는 구조라는 것을 이해하게 되었다.

---

## 한 문장으로 정리

> LLM은 Tool을 실행하지 않는다.
>
> LLM은 어떤 Tool을 어떤 인자로 실행해야 하는지 결정한다.
>
> 실제 실행은 Runtime(Python)이 담당한다.
>
> Agent는 Reasoning → Action → Observation Loop를 반복하는 시스템이다.
